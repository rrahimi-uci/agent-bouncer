## Active Findings (original report, now resolved — see Resolution above)

### AB-001 - `POST /api/test` without explicit benchmarks skips leaderboard merge

Severity: High
Status: Reproduced on current checkout

Evidence:

- `TestConfig.benchmarks` defaults to `[]` (`src/agent_bouncer/serving/api.py:850-857`)
- `POST /api/test` only adds `--merge-scoreboard` when `cfg.benchmarks` is truthy
  (`src/agent_bouncer/serving/api.py:927-948`)
- the runner treats `benchmarks=None` / `[]` as "all benchmarks"
  (`src/agent_bouncer/training/runner.py:476-481`)
- the existing API test only covers the explicit-benchmark path
  (`tests/test_serve_api.py:546-552`)
- direct probe result: `POST /api/test` with `{"exp":"e1","device":"cpu"}` launched only
  `scripts/eval/run_testing.py --exp e1 --per-class 40 --device cpu --workers 0`
  and did not add `--merge-scoreboard` or `scripts/report/compute_curves.py`

Impact:

- API clients can run a full benchmark evaluation and get an experiment record, but never update
  `outputs/benchmark_results.json`, never dump ensemble-member prediction rows, and never rebuild
  curves
- the Workbench path usually avoids this by sending explicit benchmarks, but the API contract is
  still wrong

Fix direction:

- define benchmark-mode as `test_set is None`, not `bool(cfg.benchmarks)`
- if benchmarks are omitted, either expand to the configured benchmark list explicitly or still add
  `--merge-scoreboard` when invoking `run_testing.py`

Missing tests:

- `/api/test` with omitted `benchmarks` merges to the leaderboard
- `/api/test` with `benchmarks: []` either behaves identically or is rejected with a clear `400`

### AB-002 - invalid ensemble `scope` values are accepted and broaden to all guards

Severity: High
Status: Reproduced on current checkout

Evidence:

- `scope` is a plain `str` in both optimizer request models
  (`src/agent_bouncer/serving/api.py:319-324` and `src/agent_bouncer/serving/api.py:358-361`)
- `_optimize_pool()` treats anything except `"small"` as `None`, which means "all guards"
  (`src/agent_bouncer/serving/api.py:327-329`)
- `optimize_all_ensembles()` then persists those results into the scoreboard
  (`src/agent_bouncer/serving/api.py:364-389`)
- direct probe result: `POST /api/ensemble/optimize_all` with `{"scope":"smal"}` returned `200`
  and the response `pool` included OpenAI guards and all local guards, not the intended small-only pool

Impact:

- a typo silently broadens the optimization search space
- the typo is stateful: it can mutate the real leaderboard outputs with a different optimization
  policy than the caller intended
- this is worse than a mere validation issue because it persists wrong artifacts

Fix direction:

- use `Literal["small", "all"]` or an enum in the request model
- if "all" is intentional, document and test exactly what it includes
- reject any other value at validation time before any scoreboard mutation

Missing tests:

- invalid `scope` values return `422`/`400`
- `"all"` inclusion policy is explicit and regression-tested

### AB-003 - ensemble evaluation trusts row order and sample count only

Severity: High
Status: Reproduced on current checkout

Evidence:

- dumped prediction rows contain only `[gold, pred, score, latency_ms]`
  (`scripts/eval/run_benchmarks.py:155-162` and `src/agent_bouncer/training/runner.py:411-416`)
- `evaluate_ensemble()` checks only shared benchmark name and row-count equality
  (`src/agent_bouncer/evaluation/ensembles.py:80-103`)
- direct probe result: two guards with the same benchmark and the same row count but reversed sample
  order were accepted and produced metrics instead of being rejected

Impact:

- if two guards were scored on different cached subsets, different leakage-filtered subsets, or the
  same subset in different order, the ensemble can combine scores for different prompts as if they
  were aligned
- the resulting F1/FPR/AUC can look plausible while measuring nonsense

Fix direction:

- add a stable sample identity to every prediction row
- store subset metadata beside each benchmark dump (`per_class`, seed, cache/source fingerprint,
  leakage-filter hash)
- reject or explicitly realign only when sample identities match exactly

Missing tests:

- same row count but different sample IDs must raise
- same sample IDs in different order must raise, unless explicit realignment is implemented
- `run_benchmarks.py` and `evaluate_and_record()` must emit the same prediction schema

### AB-004 - SQLite model-store backend leaks connections

Severity: Medium
Status: Reproduced on current checkout

Evidence:

- `ModelStore` uses `with sqlite3.connect(...) as con` in `__init__`, `save`, `get`, `list`, and
  `delete` (`src/agent_bouncer/tracking/model_store.py:66-68`, `90-96`, `103-106`, `126-127`,
  `145-146`)
- in Python's sqlite3 API, that context manager manages the transaction, not the lifetime of the
  connection object
- direct probe result: repeated `save()`, `list()`, and `get()` calls emitted `ResourceWarning:
  unclosed database`
- the coverage run emitted the same warning repeatedly during `tests/test_model_store.py`

Impact:

- repeated SQLite backend use leaks file descriptors / connection objects
- this can create noisy CI, and if warnings are escalated it will fail the suite
- the default backend is `fs`, so this does not hit the default Workbench path today, but the
  optional SQLite backend is not clean or production-ready

Fix direction:

- explicitly close every SQLite connection (`contextlib.closing(...)` or `try/finally: con.close()`)
- add a regression test that fails on `ResourceWarning`

Missing tests:

- SQLite backend operations run warning-clean under `warnings.simplefilter("error", ResourceWarning)`

### AB-005 - curve resync still swallows production failures

Severity: Medium
Status: Reproduced on current checkout

Evidence:

- `_resync_curves()` shells through `runpy.run_path()` and catches both `Exception` and `SystemExit`
  (`src/agent_bouncer/serving/api.py:252-264`)
- the production path is skipped in tests when `RESULTS_JSON` is monkeypatched away from the real
  artifact (`src/agent_bouncer/serving/api.py:256-259`)
- direct probe result: monkeypatching `runpy.run_path()` to raise `SystemExit(2)` caused
  `_resync_curves()` to return silently

Impact:

- ensemble endpoints can return success while `outputs/curves.json` is stale or absent
- the current code is much better than before because `7256b04` unified AUC sources, but the
  resync mechanism is still failure-silent

Fix direction:

- move curve generation behind a callable library function
- return/log a resync status instead of swallowing every failure
- test the real resync path against temp results/prediction artifacts

Missing tests:

- production-like resync updates curves when given temp artifacts
- corrupt results input surfaces a failed resync status instead of silent success

### AB-006 - `over_refusal_aware` can silently degrade to a non-augmented dataset

Severity: Medium  
Status: Reproduced on current checkout

Evidence:

- XSTest augmentation is wrapped in `try/except Exception: pass`
  (`src/agent_bouncer/data/training_sets.py:169-179`)
- metadata written to `meta.json` does not record whether augmentation was requested or added
  (`src/agent_bouncer/data/training_sets.py:190-203`)
- direct probe result: forcing `loader("xstest")` to raise still produced a successful
  `over_refusal_aware` dataset with no augmentation fields in metadata

Impact:

- a dataset can be labeled "over_refusal_aware" while containing no XSTest augmentation at all
- experiment interpretation becomes unreliable because strategy labels stop matching the actual data

Fix direction:

- record `augmentation_requested`, `augmentation_source`, and `augmentation_added`
- fail fast or emit a surfaced warning when the strategy-specific augmentation adds zero rows

Missing tests:

- XSTest loader failure is surfaced in metadata or as a controlled error
- successful augmentation records a positive `augmentation_added` count

### AB-007 - missing train files still run the trainer and report success

Severity: Medium  
Status: Reproduced on current checkout

Evidence:

- `train_and_record()` falls back to `recs = []` when `train_data` is missing
  (`src/agent_bouncer/training/runner.py:223-240`)
- it still proceeds to build config, print a success banner, call the trainer, and record an
  experiment (`src/agent_bouncer/training/runner.py:242-274`)
- direct probe result: with a fake trainer and `train_data="missing.jsonl"`, the function printed
  `Trained distilbert`, called the trainer, and returned `n_train = 0`

Impact:

- the user-facing contract says "training ran successfully" when the source dataset does not exist
- failures move deeper into the stack than necessary, and mocked or lightweight trainer paths can
  appear to succeed on zero examples

Fix direction:

- fail fast on missing files, empty JSONL, or zero valid rows before trainer invocation

Missing tests:

- missing train file raises before trainer import/call
- empty train file raises before trainer import/call
- zero-valid-row train file raises before trainer import/call

### AB-008 - split helpers can produce zero-sized test sets on tiny data

Severity: Medium
Status: Reproduced on current checkout

Evidence:

- `train_test_split()` uses `int(len * ratio)` (`src/agent_bouncer/data/split.py:131-137`)
- `ratio_split()` does the same globally and per class (`src/agent_bouncer/data/sampling.py:108-119`)
- direct probe result: a two-record dataset with `test_ratio=0.2` produced `(train=2, test=0)` for
  both helpers

Impact:

- tiny datasets can write or propagate empty holdout sets without explicit failure at the split stage
- later code may fail much farther downstream, or worse, quietly build misleading artifacts

Fix direction:

- define a minimum split contract for small inputs
- if a non-zero `test_ratio` is requested and there are enough rows to split, do not silently return
  an empty test set

Missing tests:

- one-record, two-record, and one-per-class edge cases
- stratified splits with too-few records per class
