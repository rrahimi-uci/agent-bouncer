## Active findings

### AB-015 - duplicate prompt alignment still shifts when one member is missing a middle occurrence

Severity: High
Status: Reproduced on current workspace

Evidence:

- `src/agent_bouncer/evaluation/ensembles.py:40-52` builds keyed alignment state as
  `(sample_key, occurrence_index)`.
- `src/agent_bouncer/evaluation/ensembles.py:68-74` intersects those composite keys across
  members.
- This preserves duplicates only as long as every member retains the same duplicate-count prefix.
  If one member drops the second of three identical prompts, the third occurrence is re-labeled as
  occurrence `1`, so later duplicates shift left.
- Direct probe with synthetic rows carrying an extra debug marker produced this alignment:
  `[['A1', 'A2'], ['B1', 'B3']]`
  which means member A's second duplicate was paired with member B's third duplicate after B
  skipped the middle one.
- Current tests cover duplicate preservation and full-order reversals
  (`tests/test_ensembles_eval.py:177-209`), but not the "missing middle duplicate" case.

Impact:

- Ensemble metrics can still silently combine the wrong duplicate samples when members were scored
  on slightly different duplicate-containing subsets.
- The gold-column guard will only catch this if the shifted duplicates happen to have different
  labels; if they share the same label, the corruption survives and the metrics still look valid.

Fix direction:

- Dump a real per-example identity from the benchmark cache or source row, not just prompt text
  plus local occurrence order.
- Align by that stable example id, or reject dumps that do not provide it.

Missing tests:

- a keyed member missing the middle occurrence of a duplicate prompt is rejected or correctly
  aligned by a true sample id
- `eval_tuned()` inherits the same protection because it uses the shared aligner

### AB-016 - the PDF leaderboard drops the UI's sample-size column and mixed-size warning

Severity: Medium
Status: Reproduced on current workspace

Evidence:

- The interactive Workbench leaderboard explicitly tracks sample-size ranges and warns when rows
  were scored on different `n` values:
  `src/agent_bouncer/serving/dashboard.html:883-897`.
- The PDF report builder does not carry `n` at all:
  `src/agent_bouncer/serving/leaderboard_report.py:37-45` defines no `n` column, and
  `src/agent_bouncer/serving/leaderboard_report.py:209-216` emits a footnote with no mixed-size
  warning.
- Direct probe with mixed-`n` results produced:
  `{'has_n_header': False, 'mentions_sample_size_warning': False}`.

Impact:

- A PDF exported from the same scoreboard can hide the main caution the UI shows when rows were
  scored on different sample sizes.
- That makes the exported artifact easier to misread than the in-app leaderboard, especially after
  partial benchmark runs or trained-model rows merged at different `per_class` values.

Fix direction:

- Carry the same `n` / `n_min-n_max` aggregation into the PDF table, or at minimum emit the same
  mixed-sample-size warning the UI already shows.

Missing tests:

- `build_html()` includes an `n` column or explicit warning when rows have different sample sizes

### AB-017 - `save_model` can attach an unrelated evaluation's metrics to the wrong trained model

Severity: High
Status: Reproduced on current workspace

Evidence:

- `src/agent_bouncer/serving/api.py:538-554` loads `cfg.train_exp` and `cfg.eval_exp` separately,
  then copies whatever metrics are in `eval_exp` into the saved record with no check that the eval
  belongs to the same model, version, or training run.
- `tests/test_serve_api.py:297-314` only covers the happy path; there is no mismatch rejection
  test.
- Direct probe saved a `distilbert` SFT training record while attaching a `qwen3-0.6b` GRPO eval
  result. The stored model record came out as:
  `{'saved_base_model': 'distilbert', 'saved_technique': 'sft', 'saved_metrics_f1': 0.99, 'saved_benchmark_keys': ['xstest']}`

Impact:

- Saved-model comparisons can present the wrong benchmark metrics for the wrong model.
- Once persisted, the corrupted metadata looks normal to downstream comparison and eval-only flows,
  so the operator has no obvious signal that the saved record is invalid.

Fix direction:

- Require the eval experiment to match the selected training experiment by model key and version,
  or record an explicit train-exp linkage inside eval experiments and verify it here.
- Reject mismatches with `400` instead of silently grafting foreign metrics.

Missing tests:

- mismatched `train_exp` / `eval_exp` is rejected
- missing `eval_exp` either produces an empty-metrics save explicitly or returns a clearer error,
  depending on intended UX

### AB-018 - one malformed JSON file can take down the filesystem model-store listing path

Severity: Medium
Status: Reproduced on current workspace

Evidence:

- `src/agent_bouncer/tracking/model_store.py:138-145` blindly `json.load()`s every `*.json` file
  in the filesystem backend root.
- `src/agent_bouncer/tracking/model_store.py:115-119` does the same for `get()`.
- Direct probe with a single `broken.json` file under the store root caused
  `ModelStore(backend="fs").list()` to raise:
  `JSONDecodeError Expecting property name enclosed in double quotes...`
- `src/agent_bouncer/serving/api.py:506-523` exposes these paths directly through
  `/api/saved_models` and `/api/saved_models/{model_id}`.
- Current tests only assert that non-JSON files are ignored (`tests/test_model_store.py:100-105`);
  they do not cover malformed JSON files.

Impact:

- A single malformed or partially copied JSON file can break the saved-models API surface.
- This is especially brittle because the filesystem backend is the default backend.

Fix direction:

- Treat malformed JSON like any other unreadable artifact: skip it during listing, and return
  `None` / `404` on `get()` rather than crashing the endpoint.
- Consider logging or quarantining the bad file so operators can see that something is wrong.

Missing tests:

- malformed JSON files are skipped by `list()`
- malformed single-record files do not crash `get()` or `/api/saved_models/{id}`

### AB-019 - the new local cascade endpoint ignores a partially specified stage selection

Severity: Medium
Status: Reproduced on current working tree (local uncommitted cascade feature)

Evidence:

- `src/agent_bouncer/serving/api.py:413-418` advertises `stage1` and `stage2` as optional fields.
- `src/agent_bouncer/serving/api.py:434-442` only honors explicit stages when both are present;
  otherwise it falls through to `optimize_cascade()` and auto-picks both stages.
- There are currently no tests for the cascade path at all (`rg` over `tests/` finds none).
- Direct probe calling `build_cascade()` with `stage1='user-gate'` and no `stage2` returned
  `{'stage1': 'auto-gate', 'stage2': 'auto-filter', ...}`, which proves the provided stage was
  ignored.

Impact:

- API callers cannot pin one cascade stage and auto-pick the other, even though the request schema
  and docstring imply that omitted stages are the only ones auto-selected.
- The endpoint can persist a cascade composition different from what the caller requested.

Fix direction:

- Respect any provided stage and auto-pick only the missing peer, or reject partial stage
  specification with `400` if that mode is unsupported.

Missing tests:

- `stage1` provided and `stage2` omitted
- `stage2` provided and `stage1` omitted
- cascade endpoint happy-path coverage in `tests/test_serve_api.py`
